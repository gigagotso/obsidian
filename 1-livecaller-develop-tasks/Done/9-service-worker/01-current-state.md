# Service worker — current state (verified in code, 2026-06-10)

Everything below was verified directly in the repo (`www/app`, branch `develop`, app version `2.12.1`).

| Piece | Where | What it does today |
|---|---|---|

## What the generated service worker does (all defaults)

Because `workboxOptions` is empty, the generated `service-worker.js`:

- **Precaches the entire build output including `index.html`** into Cache Storage.
- Serves precached files cache-first — **the network is never consulted for `index.html`**.
- Has **`skipWaiting: false` and `clientsClaim: false`** (Workbox 4 defaults) — a newly
  installed SW parks in `waiting` until *every* tab is closed. A refresh does not count.
- Has **no runtime caching routes** — API calls and anything not in the precache manifest pass
  straight through to the network (good; API responses are not cached).
- Has **no `message` listener** — there is no way to tell a waiting SW to `skipWaiting()` from
  the page.

## The resulting bug (recap)

1. Deploy ships new `index.html` + new hashed bundles.
2. The old SW intercepts navigation and serves the **old precached `index.html`** → old hashes →
   old app. The server is never asked.
3. The browser does detect the changed `service-worker.js` and installs the new SW, but it sits
   in `waiting` forever because tabs stay open.
4. `updated()` fires in `registerServiceWorker.js` — and only logs
   `"New content is available; please refresh."` to the console. **This is the exact hook the
   update-notification feature should use** — the detection mechanism already works, it's just
   wired to a no-op.
5. User is stuck on the old version until they clear site data or close all tabs.

## Observed case: hard reload shows the new version — second refresh goes BACK a version

Reproduced symptom: `Cmd/Ctrl+Shift+R` (Hard Reload) or right-click reload → "Empty Cache and
Hard Reload" loads the **new** version… and then one ordinary refresh later the app is the
**previous** version again. This looks insane ("my app time-traveled backwards") but it is
spec-correct behavior, and it's strong confirmation that the service worker — not HTTP/CDN
caching — is the culprit:

1. **Hard reload bypasses the service worker, it does not remove it.** A shift-reload
   navigation is loaded with a "force refresh" flag: the browser skips the SW interception
   entirely and goes straight to the network (also skipping HTTP cache). The page you get is
   **uncontrolled** — `navigator.serviceWorker.controller === null` for that load. Fresh
   `index.html` → fresh hashes → new app. So far so good.
2. **"Empty Cache and Hard Reload" empties the wrong cache.** It clears the **HTTP cache** for
   the site, but it does **not** touch **Cache Storage** and does **not** unregister the SW.
   The old precached `index.html` is still sitting on disk, and the old SW is still registered.
   (Only DevTools → Application → "Clear site data", or browser settings → clear site data,
   removes Cache Storage + the registration.)
3. **The next normal refresh is intercepted again.** A plain reload is a regular navigation, so
   the still-registered old SW becomes the controller again and answers from its precache — the
   **old** `index.html`, old hashes, old app. The app "downgrades" one version before your eyes.
4. Meanwhile the new SW that was fetched during all this is still parked in `waiting` (tabs
   never all closed), so nothing ever promotes the new version permanently.

In short: hard reload is a **one-load exemption** from the SW, not a fix. The only user-side
escapes remain "Clear site data" or closing every tab. After the planned fix
([[03-short-term-plan-update-notification]]) this whole class of behavior disappears: normal
refreshes fetch HTML from the network, so there is no stale copy to fall back to.

## Constraints to keep in mind

- **The SW must stay** — LiveCaller is installed as a desktop PWA and Chromium requires a
  registered service worker for installability.
- `publicPath` comes from `VUE_APP_ASSETS_URL` and the SW registers at `BASE_URL` — the SW must
  remain same-origin with the app for registration to work (it does today).
- Updates are only *detected* when the browser re-checks `service-worker.js` (on navigation and
  at most every 24h) — so `service-worker.js` and `index.html` must be served with
  `Cache-Control: no-cache` at the HTTP/CDN layer for detection to be prompt.
